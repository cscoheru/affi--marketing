# 会话协作框架

**版本**: 1.0.0
**创建日期**: 2026-03-03
**适用范围**: Affi-Marketing 项目所有会话

---

## 🎯 协作目标

建立灵活、动态的会话间通信机制，确保：
1. 每个会话可独立工作
2. 必要时能与其他会话沟通
3. 状态变更及时同步
4. 依赖关系清晰可见

## 🔄 会话通信协议

### 1. 状态广播

**触发条件**:
- 会话开始/结束
- 里程碑完成
- 阻塞问题出现
- API 接口变更

**广播方式**:
```markdown
在 `docs/SESSION_LOG.md` 中添加：

## [会话ID] 状态更新

**时间**: 2026-03-03 14:30
**状态**: 完成任务 X
**影响**: [列出受影响的其他会话]
**下一步**: [即将开始的任务]
**需求**: [需要其他会话配合的事项]
```

### 2. 直接消息传递

**场景**: 需要特定会话响应

**方式**: 在 `docs/MESSAGES_{目标会话ID}.md` 中留言

```markdown
# 来自 S01-FE 的消息

**时间**: 2026-03-03 14:30
**主题**: API 接口变更请求

## 内容
需要修改 /api/v1/experiments 接口，添加过滤参数...

## 期望响应
- [ ] 确认是否可行
- [ ] 预计完成时间
- [ ] 需要的配合

## 处理状态
- [ ] 待处理
- [x] 已处理
- [x] 已关闭
```

### 3. 代码共享契约

**API 接口变更流程**:

```
[发起会话]
    │
    ├─→ 1. 在 `docs/API_CHANGES.md` 提出变更
    │
    ├─→ 2. 等待依赖会话确认 (24h)
    │
    ├─→ 3. 实施变更
    │
    └─→ 4. 更新 `docs/api/spec.json`
```

**数据结构变更流程**:

```
[发起会话]
    │
    ├─→ 1. 在 `docs/SCHEMA_CHANGES.md` 提出变更
    │
    ├─→ 2. 更新 `docs/DATABASE_SCHEMA.md`
    │
    ├─→ 3. 创建迁移脚本 `migrations/XXX_*`
    │
    └─→ 4. 通知依赖会话
```

## 🚨 冲突解决

### API 接口冲突

**场景**: 前后端对同一接口理解不一致

**解决流程**:
1. 在 `docs/API_CHANGES.md` 创建提案
2. 双方会话在各自会话中讨论
3. 通过 `MESSAGES_*.md` 达成一致
4. 更新 `docs/API_SPEC.md`

### 数据结构冲突

**场景**: 数据库表结构与代码不匹配

**解决流程**:
1. 在 `docs/SCHEMA_CHANGES.md` 创建提案
2. 数据库工程师会话评审
3. 创建迁移脚本
4. 通知所有依赖方

### 优先级冲突

**场景**: 多个会话同时要求同一资源

**解决流程**:
1. 查看会话注册表中的优先级
2. 项目经理会话 (PM) 协调
3. 更新任务优先级
4. 重新分配资源

## 🔄 协作模式

### 模式 1: 独立工作 (默认)

```
[会话A] ────────> [产出代码]
                      │
                      ▼
                [Git 仓库]
                      │
                      ▼
                [会话B] 读取
```

### 模式 2: 依赖协作

```
[会话A] 完成 ──> [通知依赖方] ──> [会话B] 开始
     ▲                                    │
     └──────────── [等待完成] ──────────────┘
```

### 模式 3: 并行协作

```
[会话A] 和 [会话B] 同时开发
     │              │
     ├─> 定期同步 ──┤
     │              │
     └─> 合并代码 <─┘
```

### 模式 4: 实时协作

```
[会话A] ──> [共享文档] <── [会话B]
              (实时编辑)
```

## 📊 协作检查清单

### 会话开始时
- [ ] 查看 `docs/SESSION_REGISTRY.md` 了解其他会话状态
- [ ] 检查 `docs/MESSAGES_{当前会话ID}.md` 是否有未读消息
- [ ] 查看 `docs/API_CHANGES.md` 和 `docs/SCHEMA_CHANGES.md`

### 任务完成时
- [ ] 更新 `docs/SESSION_LOG.md` 记录完成状态
- [ ] 如果影响其他会话，发送消息通知
- [ ] 如果有 API/Schema 变更，提交变更提案

### 遇到阻塞时
- [ ] 在 `docs/SESSION_LOG.md` 记录阻塞问题
- [ ] 向相关会话发送消息求助
- [ ] 如果无法解决，上报项目经理会话

## 🔔 通知机制

### 自动通知 (通过文件变更)

| 文件变更 | 通知对象 | 内容 |
|----------|----------|------|
| `docs/API_CHANGES.md` | 所有会话 | API 有新变更提案 |
| `docs/SCHEMA_CHANGES.md` | 所有会话 | 数据结构有新变更 |
| `docs/MESSAGES_{ID}.md` | 目标会话 | 有新消息 |
| `docs/SESSION_LOG.md` | 所有会话 | 会话状态更新 |

### 手动通知

**在会话中执行**:
```
请通知 [目标会话ID]: [消息内容]
```

Claude Code 会将消息写入 `docs/MESSAGES_{目标会话ID}.md`

## 📝 协作最佳实践

### DO ✅
1. **及时更新状态** - 每完成一个里程碑更新 SESSION_LOG
2. **提前沟通** - API 变更提前 24h 通知
3. **文档先行** - 变更先写文档，再写代码
4. **尊重依赖** - 等待依赖方完成后再开始

### DON'T ❌
1. **不要** 直接修改他人代码
2. **不要** 在不通知的情况下变更 API
3. **不要** 忽略其他会话的消息
4. **不要** 阻塞其他会话超过 24h
